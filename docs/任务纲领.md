# LunaCane 智能拐杖任务纲领

## 一、项目定位

LunaCane 面向行动不便、独居或需要看护的老人，目标是在拐杖形态上集成两类核心能力：

1. 可对话 AI 助手：老人按键录音后，拐杖把语音上传到服务器，服务器完成语音识别、大模型问答和语音合成，再把回答下发给 ESP32 播放。
2. 跌倒检测：拐杖通过 BMI270 采集加速度和角速度数据，训练轻量级 TinyML 模型，后续部署到 ESP32 侧实现本地跌倒识别与告警。

当前仓库已经具备从硬件采集、服务端处理到模型训练的基础链路。后续任务重点不是重新起项目，而是把已有链路整理成稳定、可演示、可验收的一体化系统。

## 二、当前仓库能力盘点

### 1. 语音对话链路

- `hardware/VoiceAssistant/VoiceAssistant.ino`：ESP32 端 I2S 麦克风录音、按键消抖、音频上传、回答音频下载与 I2S 播放。
- `voice/main_server.py`：FastAPI 服务，提供 `/upload_audio` 和 `/get_audio/{filename}` 接口。
- `voice/luna_core.py`：串联阿里云 NLS ASR、兼容 OpenAI Chat Completions 的大模型接口、edge-tts 和 ffmpeg 音频转换。
- `.env.example`：整理了语音服务需要的模型、阿里云和 ffmpeg 配置项。

### 2. 跌倒检测链路

- `hardware/CollectData/CollectData.ino`：ESP32 + BMI270 以 50Hz 采样，批量上传 IMU 数据。
- `data_collection/data_server_labeled.py`：带动作标签的数据采集服务，保存 `data/raw/bmi_data_<label>.csv`。
- `data_collection/data_server_debug.py`：无标签调试采集服务，支持采集上限，避免调试数据无限增长。
- `ml/preprocess.py`：合并 CSV、构造特征、标准化、滑动窗口切分、训练 1D-CNN，并导出 Keras、TFLite 和 scaler 参数头文件。

### 3. 调试经验沉淀

- `docs/故障排查记录.md` 已记录 WiFi、BMI270、供电 brownout、I2C 稳定性和采集上限等问题，对后续硬件联调有直接参考价值。

## 三、阶段目标

### 阶段 1：完成稳定演示闭环

目标是让项目能够在现场稳定展示，不依赖临时调试经验。

- 语音助手可以完成“按住说话、松开发送、播放回答”的完整闭环。
- 跌倒检测可以完成“采集动作数据、训练模型、导出模型文件”的完整闭环。
- README 和文档说明能让组员按步骤复现采集、训练、语音服务和硬件烧录。
- 明确哪些配置必须本地填写，哪些产物不应提交到仓库。

验收标准：

- ESP32 能成功连接 2.4GHz WiFi，并访问电脑局域网服务。
- `/upload_audio` 能收到音频并返回可播放的回答文件名。
- `data_collection/data_server_labeled.py` 能稳定生成带标签 CSV。
- `ml/preprocess.py` 在已有数据集上能完成训练并导出 `models/lunacane_model.tflite` 与 `models/scaler_params.h`。

### 阶段 2：提升跌倒检测可靠性

目标是从“能训练”推进到“可信地检测跌倒”。

- 规范采集标签，至少覆盖 `stand`、`walk`、`sit_down`、`put_down`、`tap`、`fall` 等动作。
- 每类动作由多名成员、多种姿态、多次重复采集，避免模型只记住单人或单场景特征。
- 建立固定评测集，单独记录召回率、误报率、混淆矩阵和典型误判样本。
- 根据测试结果调整窗口大小、步长、跌倒阈值和模型结构。
- 将 `scaler_params.h` 和 TFLite 模型部署到 ESP32 端，形成边采样边推理的固件版本。

验收标准：

- 跌倒召回率优先达标，尽量减少漏报。
- 正常拐杖动作、敲击、放下、坐下等场景误报率可接受。
- 能在串口或服务端日志中清楚看到每次窗口推理结果。
- 形成一份模型评测记录，说明数据量、标签分布、指标和失败案例。

### 阶段 3：完成硬件一体化

目标是把语音助手和跌倒检测放进同一套智能拐杖原型。

- 统一 ESP32 引脚规划，避免 I2S、BMI270、按键、喇叭和供电线路互相冲突。
- 明确电池、升压、充电保护、电容滤波、开关和外壳布局。
- 处理 WiFi 发包导致的供电波动，降低 brownout 和 BMI270 读失败概率。
- 设计跌倒告警策略：本地蜂鸣、语音提示、上传事件或后续接入短信/微信通知。
- 保留调试串口输出，但演示版本应减少刷屏日志，突出关键状态。

验收标准：

- 电池供电下连续运行，不因 WiFi 发射频繁重启。
- 语音录音、语音播放和 IMU 采样不会互相阻塞到不可用。
- 跌倒事件能够触发明确的本地或服务端告警。
- 形成接线图、实物照片和常见故障处理说明。

### 阶段 4：准备结题材料和演示

目标是把工程成果转换为大创结题需要的材料。

- 整理项目背景、需求分析、总体架构、硬件设计、软件设计和算法设计。
- 汇总数据采集流程、模型训练流程、评测指标和实验结果。
- 准备演示脚本：正常问答、日常动作不误报、模拟跌倒触发告警。
- 准备答辩材料：创新点、难点、已解决问题、剩余不足和后续改进。

验收标准：

- 仓库文档能支撑他人复现实验。
- PPT、演示视频、测试数据摘要和关键代码截图完整。
- 对语音链路、跌倒算法和硬件稳定性都有可解释的结果。

## 四、近期任务清单

### 语音助手

- 补充服务器启动前检查，启动失败时明确提示缺少的 `.env` 配置或 ffmpeg。
- 固化 ESP32 与服务器的音频格式：16kHz、单声道、16bit PCM/WAV。
- 增加基础错误处理：ASR 无结果、大模型请求失败、TTS 转换失败、音频文件不存在。
- 设计适合老人使用的系统提示词，回答保持短句、口语化、少符号。
- 增加常用离线兜底语音，例如“网络不稳定，请稍后再试”。

### 跌倒检测

- 固定采集规范：每次采集前记录标签、采集人、动作说明、时长和设备佩戴/安装位置。
- 增加数据质检脚本，检查空值、采样间隔、标签分布和异常尖峰。
- 拆分训练集、验证集和测试集，避免同一次连续采集的数据同时出现在训练和测试中。
- 记录每次模型训练的参数、数据版本和评测结果。
- 推进 TFLite Micro 或 ESP32 端推理集成，把模型真正跑到拐杖上。

### 硬件与系统联调

- 完成最终引脚表，标清 BMI270、I2S 麦克风、I2S 喇叭、按键、电源和预留接口。
- 按故障排查记录补齐电容、短粗线和稳定升压模块。
- 统一端口和 IP 配置说明，减少每次换网络后的手动排错成本。
- 明确语音服务和数据采集服务是否共用端口；如需同时运行，应拆分端口。
- 设计跌倒后的人机交互流程，例如先语音询问“是否需要帮助”，无响应再告警。

### 文档与管理

- README 保持快速上手，`docs/` 放详细任务、排错、评测和结题材料。
- 所有密钥、音频、原始数据、模型产物继续通过 `.gitignore` 排除。
- 每个阶段结束后补充对应截图、日志、测试结果和问题记录。
- Pull Request 说明应写清楚改动目的、完成内容、验证方式和剩余风险。

## 五、分工建议

- 硬件组：ESP32 接线、供电、电池、外壳、传感器安装、烧录与现场联调。
- 语音组：FastAPI 服务、ASR/LLM/TTS 配置、音频格式、回答策略和异常兜底。
- 算法组：数据采集规范、特征工程、模型训练、评测指标、TFLite/ESP32 推理。
- 文档组：README、任务记录、故障记录、评测报告、PPT 和演示脚本。

各组之间需要固定接口：

- 语音接口：ESP32 上传 WAV，服务器返回回答音频文件名，ESP32 下载 PCM 流播放。
- 跌倒数据接口：ESP32 上传 `t/ax/ay/az/gx/gy/gz`，服务端补充合成特征和标签。
- 模型部署接口：训练脚本导出 TFLite 模型和 scaler 参数，硬件端按同样特征顺序归一化。

## 六、主要风险与应对

- 供电不稳：优先按故障排查记录补齐电容、升压模块和低内阻电池，避免 WiFi 发包导致重启。
- 数据不足：跌倒样本少会导致模型指标虚高或泛化差，必须多成员、多动作、多场景采集。
- 误报与漏报：跌倒检测要优先降低漏报，同时通过动作类别扩充和阈值调整控制误报。
- 网络依赖：语音助手依赖局域网、外部 ASR、LLM 和 TTS 服务，演示前必须准备备用热点和兜底提示。
- 端口冲突：语音服务和采集服务默认都可能使用 8000 端口，同时演示时需要提前规划。
- 隐私与安全：不要提交 `.env`、老人录音、原始敏感数据或云服务密钥。
