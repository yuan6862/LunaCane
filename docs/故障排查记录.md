# 故障排查记录

> 记录 ESP32（BMI270 IMU）+ Flask 数据采集链路调试中遇到的故障与解决方案。
> 链路：`CollectData.ino`（BMI270，50Hz，每 20 条批量上传）→ POST `/sensor` → `data_collection/data_server.py` → `data/raw/*.csv`

---

## 2026-06-06

### 1. 服务器 host 写死成非本机 IP，Flask 起不来 / ESP 连不上

**现象**：`data_server.py` 里 `app.run(host="192.168.x.x")`，而该 IP 不一定是本机当前地址；手机热点动态分配，IP 会变化。host 写死成非本机 IP 时 Flask 无法绑定。

**解决**：
- 服务器固定用 `app.run(host="0.0.0.0", ...)`，监听所有网卡，IP 怎么变都能绑定。
- ESP 端 `serverUrl` 仍需填**电脑当前的局域网 IP**。测试前用 `Get-NetIPAddress -InterfaceAlias WLAN` 确认，IP 变了要同步改 ESP 并重新烧录。

### 2. `BMI270 read failed` 反复出现

**现象**：串口反复打印 `BMI270 read failed`（loop 里 `imu.getSensorData() != BMI2_OK`）。

**根因定位**（追 SparkFun_BMI270 库源码）：
- 失败最终来自 I2C 读寄存器地址时收到 NACK → 库返回 `BMI2_E_COM_FAIL`（= -2）。
- `initIMU()` 能成功（否则会卡在 `while(1)` 打印 init failed），说明上电瞬间 I2C 是通的、接线/地址/型号都没问题。
- "init 能成、连续读却失败" + "失败成簇出现后又自己恢复" → **瞬态干扰**，不是硬件彻底坏。

**真凶**：WiFi 发射瞬间电流尖峰（峰值数百 mA、持续几十 µs~几 ms）把 3.3V 轨瞬间拉到欠压门限附近，BMI270 那几次读不应答；发包结束电压回升即恢复。失败节奏与每 ~400ms 的批量上传、以及刚连 WiFi 那会儿相关。能"干净恢复"说明芯片只是瞬时欠压、没真正掉电复位（否则会丢配置）。

### 3. 电池供电时连不上 / 间歇掉线（brownout 实锤）

**现象**：拔掉 USB 改电池供电后，ESP "连一下→发一小批→掉线→一段时间不连热点"。服务器侧表现为 CSV 行数偶尔涨一批（如 +201）然后长时间不动。

**根因**：电池电量不足 / 内阻大，扛不住 WiFi 发射电流尖峰 → 3.3V 跌破欠压门限 → ESP 复位或断开关联。

**验证**：给电池充满电后再测 → 数据稳定连续上传（服务器日志每 ~1s 一次 `POST /sensor 200`，无掉线）。根因确认为供电不足。

### 4. 关键认知：万用表显示 5.21V，为什么还会供电不稳？

测电源输入显示 5.21V，仍出现 brownout，原因：
- **万用表测的是平均值**，每秒才采样几次，抓不到几毫秒的 WiFi 电流尖峰造成的瞬态电压跌落（要示波器才看得到）。
- **出问题的是 3.3V 轨**，不是 5V 输入。5V 经板载 LDO（AMS1117 类）降到 3.3V，LDO 瞬态响应有限，负载突变时 3.3V 会下垂；ESP32 欠压检测在 3.3V 轨上，瞬时跌破门限就复位。
- **电压够 ≠ 电流够**。5.21V 不代表电源能瞬间吐出尖峰电流；电池没电时内阻大，尖峰电流下压降大（V=开路−I×内阻）。充满电内阻变小，瞬态压降不足以触发门限，所以就稳了。
- **测量点**：要把表笔顶在芯片 VCC/GND 引脚上、且在发包运行时测，才反映真实；测在电源端子会漏掉细长导线的 IR 压降。

### 解决方案汇总（按性价比）

**硬件（治本）**
1. 就近并**去耦电容**：ESP 的 3V3-GND、BMI270 的 VCC-GND 各并 `100µF 电解 + 0.1µF 陶瓷`，用电容顶住瞬态尖峰。
2. 用**短而粗**的电源/地线，减小 IR 压降。
3. 选**低内阻、放电能力强**的电池；用升压模块要选大电流型号。
4. SDA/SCL 各加 `4.7kΩ` 上拉到 3.3V。

**软件（兜底，写进 `CollectData.ino`）**
```cpp
// connectWiFi() 内 WiFi.begin() 之后：
WiFi.setSleep(false);                 // 关 modem 省电，连接更稳
WiFi.setAutoReconnect(true);          // 断线自动重连
WiFi.setTxPower(WIFI_POWER_8_5dBm);   // 降发射功率→减小电流尖峰→减少 brownout

// sendBatch() 断网检查改成主动重连：
if (WiFi.status() != WL_CONNECTED) { WiFi.reconnect(); return; }
```

**其它**
- 检查手机热点设置，关掉"无设备时自动关闭热点 / 踢空闲客户端"。

### 5. debug 服务器加采集上限

`data_server.py` 增加 `MAX_SAMPLES = 3000`（50Hz≈60s），写满后只回 `{"status":"full"}` 不再落盘，避免 debug 测试时 CSV 无限增长。注意计数是**进程内**的，重启服务器归零；想从干净文件开始要先删 `data/raw/bmi_data_debug.csv`。

### 6. 供电方案与采购清单

当前供电链路：`18650 → TP4056(充电) ; 18650 → 升压模块 → 5V → ESP`。
注意 TP4056 的 `OUT+/OUT-` 走的是保护电路(DW01+FS8205)输出，WiFi 尖峰电流穿过保护 MOS 会有额外压降、甚至可能触发过流误断 → 加重 brownout。需靠缓冲电容/更强供电改善。

**采购清单（按优先级）**

| 优先级 | 物品 | 规格/型号建议 | 数量 | 用途 |
|--------|------|--------------|------|------|
| ★★★ | 电解电容 | 470µF~1000µF / 6.3V 以上 | 2~3 | 并在**升压模块输出端 5V-GND**，顶 WiFi 发射尖峰（治本关键） |
| ★★★ | 陶瓷电容 | 0.1µF (104) | 5~10 | 与电解并联滤高频；就近并 ESP 3V3-GND、BMI270 VCC-GND |
| ★★☆ | 电解电容 | 100µF~470µF | 2 | 并在**升压模块输入端 / TP4056 OUT 侧**，缓冲抽流 |
| ★★☆ | 升压模块 | 大电流型（额定 ≥2A，避免廉价 MT3608 小板） | 1 | 替换瞬态纹波大的劣质升压模块 |
| ★★☆ | 18650 电芯 | **带保护板、低内阻、放电能力强**（标 ≥2A 持续） | 按需 | 电芯内阻大是 brownout 主因之一 |
| ★☆☆ | TP4056 模块 | **带保护版**(TP4056+DW01+FS8205)，或一体化充放电+升压模块(如 134N3P) | 1 | 充电+保护；一体模块可边充边用、省接线 |
| ★☆☆ | 上拉电阻 | 4.7kΩ | 2 | SDA/SCL 各上拉到 3.3V，改善 I2C 时序裕量（可选） |
| ★☆☆ | 导线/接插件 | 短而粗的硅胶线、可靠杜邦/端子 | 若干 | 减小线阻与接触电阻造成的 IR 压降 |

> 核心思路：**电容（就近缓冲尖峰）+ 强放电电池/升压（降低源内阻）+ 短粗线（降线阻）**，三者一起把 WiFi 尖峰造成的瞬态电压跌落压在欠压门限之上。

> ⚠️ 排错口诀：**USB 正常、电池就掉线 = 八成是供电 brownout，别信万用表的平均读数。**
